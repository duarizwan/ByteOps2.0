# Design: Action Center — Runs Page Redesign

**Date**: 2026-05-31
**Feature**: Action Center (replaces `/runs` split layout)
**Status**: Approved
**Scope**: `frontend/src/components/runs/`, `backend/app/api/chat.py`, orchestrator clarification logic

---

## Problem

The current `/runs` page is a developer debugging tool masquerading as a user feature. A ReactFlow DAG of execution nodes (Intent Router, Agent Plan, Platform API) is the first thing a user sees when they want to know "what did the AI do?" Most users can't read it and don't need to.

The page also has no approval UI — runs with `waiting_approval` status have nowhere to be actioned from the UI.

---

## Goal

Make the runs page genuinely useful for regular users:
- Plain English summary of every action the AI took on their behalf
- Pending approvals surfaced prominently with inline Approve / Cancel
- Failed actions grouped separately with fix CTAs
- Graph preserved but demoted to an on-demand drill-down

---

## Architecture

### Layout change

**Before:** Split layout — list panel (left 280px) + GraphCanvas (right, full height)

**After:** Single-column Action Center feed, full width. GraphCanvas opens in a slide-in drawer (400px, right side) when "trace →" is clicked.

### Route

`/runs` — unchanged. Page title and nav label change to "Action Center".

### Component structure

```
ActionCenter (replaces RunsPage)
├── ActionCenterTopBar       — ← Dashboard | title | filter tabs (All/Pending/Failed)
├── AwaitingApprovalSection  — amber cards, pinned, hidden when empty
├── NeedsFixingSection       — red cards, pinned, hidden when empty
├── ActionFeed               — date-grouped completed action cards
│   ├── DateSeparator
│   └── ActionCard × N
└── TraceDrawer              — slide-in panel containing existing GraphCanvas
```

### Data flow

```
useAgentRuns() → hasCrudOperation filter → ActionCenter state
                                               ├── pending = runs where status === "waiting_approval"
                                               ├── failed  = runs where status === "failed"
                                               └── history = runs where status === "completed" | "cancelled"
```

## Components

### ActionCard

Each completed or cancelled action displays:

```
[3px status bar] [platform badge] [natural language summary]     [time]
                 [detail line: key secondary field]          [trace →]
```

- **Status bar**: green = completed, grey = cancelled
- **Platform badge**: coloured pill (Gmail, Jira, Slack, Cal, GitHub, Dropbox)
- **Summary**: generated by `summarizeAction(run)` (see below)
- **Detail line**: most useful secondary field (subject, project key, channel, filename)
- **trace →**: opens TraceDrawer for this run

### AwaitingApprovalCard

```
[3px amber bar] [platform badge] [what it's about to do]         [time]
                [consequence line — "will send to 5 people"]
                [Approve]  [Edit in chat →]  [Cancel]
```

- **Approve** → `POST /api/agent-runs/{id}/approve`
- **Cancel** → `POST /api/agent-runs/{id}/cancel`
- **Edit in chat →** → navigates to `/dashboard` (the conversation that triggered the run)

### NeedsFixingCard

```
[3px red bar] [platform badge] [what failed]                     [time]
              [error reason — human readable]          [fix CTA →]
```

Fix CTAs by failure type:
- Auth error (detected by `run.error` containing "401", "403", "token", "expired", or "unauthorized") → "Reconnect {platform} →" (links to `/settings`)
- Timeout / API error → "Retry →" (calls `POST /api/agent-runs/{id}/retry`)
- Unknown → no CTA, just error text

### TraceDrawer

- Slides in from right, 400px wide, full viewport height
- Header: run summary + X close button
- Body: existing `GraphCanvas` component, unchanged
- Overlay backdrop closes drawer on click

### FilterTabs

Three tabs in top bar: **All** (default) · **Pending** · **Failed**
- All: shows AwaitingApproval + NeedsFixing sections + full history
- Pending: shows only AwaitingApproval section
- Failed: shows only NeedsFixing section + failed history items

## Natural Language Summaries

`summarizeAction(run: AgentRun): { summary: string; detail: string }` — pure function, no side effects.

Looks for the first CRUD tool_call step and extracts from its `input` payload:

| Intent | Tool name | Summary | Detail |
|--------|-----------|---------|--------|
| `gmail` | `send_email` | "Sent email to {to}" | "Subject: {subject}" |
| `gmail` | `reply_to_email` | "Replied to {thread}" | "To: {to}" |
| `gmail` | `trash_email` | "Trashed email: {subject}" | "{count} message(s)" |
| `gmail` | `create_draft` | "Saved draft for {to}" | "Subject: {subject}" |
| `calendar` | `create_event` | "Created event: {title}" | "{date} · {attendees} people" |
| `calendar` | `update_event` | "Updated event: {title}" | "{what changed}" |
| `calendar` | `delete_event` | "Deleted event: {title}" | "{date}" |
| `slack` | `send_message` / `post_message` | "Posted to #{channel}" | "{first 60 chars of message}" |
| `jira` | `create_issue` | "Created {type} {key}" | "{title}" |
| `jira` | `update_issue` | "Updated {key}" | "{what changed}" |
| `jira` | `add_comment` | "Commented on {key}" | "{first 60 chars}" |
| `github` | `merge_pull_request` | "Merged PR #{number}" | "{title}" |
| `github` | `create_issue` | "Created issue #{number}" | "{title}" |
| `dropbox` | `upload_file` | "Uploaded {filename}" | "to {path}" |
| `dropbox` | `delete_file` | "Deleted {filename}" | "{path}" |
| `dropbox` | `move_file` | "Moved {filename}" | "to {destination}" |
| fallback | any | "{intent} action completed" | "{tool name humanized}" |

All fields are extracted from `step.input` — if a field is missing, it falls back gracefully to a shorter string.

## Intent Clarification (Orchestrator)

When the orchestrator detects **doubtful intent**, it returns a clarification response to the user in chat **before creating an agent run**.

### What counts as doubtful

- Confidence score below threshold (orchestrator internal)
- Message maps to multiple platforms ("send a message" → Slack or Gmail?)
- Target is ambiguous ("post an update" — which channel / project?)
- Destructive action phrased casually ("clean up my calendar", "remove the old tickets")

### Clarification response format (in chat)

> "Just to confirm — I'll **{plain English action}**. Is that right?"
> `[Yes, go ahead]` `[No, let me rephrase]`

Examples:
> "Just to confirm — I'll **post a message in #general on Slack**. Should I proceed?"
> "To confirm — I'll **create a bug ticket** titled 'Login redirect fails on mobile' in PROJ. Go ahead?"

### Behaviour

- No run is created until the user confirms
- `[Yes, go ahead]` → orchestrator proceeds, creates the run normally
- `[No, let me rephrase]` → no run created, user can restate their request
- This is entirely in-chat — no new backend status, no Action Center entry

## Backend Changes

Two new endpoints required:

### `POST /api/agent-runs/{id}/approve`
- Auth: Bearer token (Clerk)
- Sets run status from `waiting_approval` → `running`, resumes execution
- 404 if run not found or not owned by current user
- 409 if run is not in `waiting_approval` status

### `POST /api/agent-runs/{id}/cancel`
- Auth: Bearer token (Clerk)
- Sets run status → `cancelled`
- 404 / 409 same as above

### `POST /api/agent-runs/{id}/retry` *(stretch)*
- Re-triggers a failed run
- Can be deferred to a follow-up

## Success Criteria

- A user can understand what the AI did in under 5 seconds without reading a graph
- Pending approvals are impossible to miss (pinned, amber, visible immediately on page load)
- Failed actions with auth errors link directly to the settings reconnect flow
- The graph is still accessible for any run via "trace →"
- Intent clarification prevents the AI from executing ambiguous destructive actions without confirmation

---

## Phases

**Phase 1 (this spec):** Frontend Action Center redesign + approve/cancel backend endpoints. Ships independently.

**Phase 2 (follow-up):** Intent clarification in the orchestrator. Requires changes to `orchestrator.py` and chat response format. Separate spec.

- Push notifications for pending approvals (future feature)
- Run retry endpoint (stretch — can ship without it)
- Mobile drawer variant (can be added in polish pass)
